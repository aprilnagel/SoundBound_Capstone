Readers represent the core user role in your app. They can search for books, create playlists, manage their account, and explore titles by an author. This flow outlines every action a Reader can take from onboarding to playlist management.

1. Account Access
Readers must authenticate before using the app.
1.1 Sign Up / Log In
Goal: Access the app.
Flow:
- User opens the app
- User signs up or logs in
- Backend authenticates credentials
- User is redirected to the home/search screen

2. Book Discovery
Readers search for books to begin playlist creation.
2.1 Search for a Book
Goal: Find the correct book.
Flow:
- User enters a search term (title or author)
- App calls the book search endpoint
- Backend returns clean results containing:
- title
- author_name
- cover_url
- api_id
- optional: first_publish_year, isbn[]
- User scrolls and selects a book

2.2 View Book Details
Goal: Confirm the book and begin playlist creation.
Flow:
- App requests full book details
- Backend returns:
- title
- description
- subjects (tags)
- cover_url
- User views the book detail screen
- User selects ‚ÄúCreate Playlist‚Äù

3. Song Discovery
Readers search for songs to add to their playlist.
3.1 Search for Songs
Goal: Find songs by title, artist, or mood.
Flow:
- User enters a search term
- App calls one of the music search endpoints:
- search by title
- search by artist
- search by tags (mood/genre)
- Backend returns clean song results containing:
- title
- artist_name
- album_art_url
- spotify_id
- User selects songs to add

4. Playlist Creation
Readers build and save playlists for a selected book.
4.1 Build Playlist
Goal: Assemble the playlist.
Flow:
- User adds songs to the playlist draft
- User can reorder or remove songs
- User enters playlist title and optional description
- User selects ‚ÄúSave Playlist‚Äù

4.2 Save Playlist
Goal: Store the playlist in the user‚Äôs account.
Flow:
- App sends playlist data to backend
- Backend creates playlist record linked to:
- user_id
- book_id
- playlist_songs
- Backend returns success
- User sees confirmation

5. Playlist Management
Readers can view, edit, and delete their playlists.
5.1 View My Playlists
Goal: Access previously created playlists.
Flow:
- User opens ‚ÄúMy Playlists‚Äù
- App requests user‚Äôs playlists
- Backend returns list of playlists
- User selects a playlist to view

5.2 Edit or Delete Playlist
Goal: Manage existing playlists.
Flow:
- User opens a playlist
- User chooses to:
- add/remove songs
- reorder songs
- rename playlist
- delete playlist
- Backend updates or deletes the playlist
- User sees updated playlist

6. Author Title Browsing
Readers can explore more books by the same author.
6.1 View All Titles by an Author
Goal: Discover additional books.
Flow:
- User clicks an author name on a book
- App calls book search filtered by author
- Backend returns list of titles by that author
- User selects a book and continues the flow

7. Account Management
Readers can manage their account at any time.
7.1 Update Account Information
Goal: Keep account info accurate.
Flow:
- User opens ‚ÄúAccount Settings‚Äù
- User updates name, email, or password
- Backend validates and saves changes
- User sees confirmation

7.2 Change Password
Goal: Update password securely.
Flow:
- User enters current + new password
- Backend verifies and updates password
- User sees confirmation

7.3 Password Reset (Forgot Password)
Goal: Recover account access.
Flow:
- User selects ‚ÄúForgot Password‚Äù
- User enters email
- Backend sends reset link
- User sets new password
- User logs in

7.4 Delete Account
Goal: Remove user data.
Flow:
- User selects ‚ÄúDelete Account‚Äù
- User confirms
- Backend deletes:
- user
- user playlists
- User is logged out

7.5 Log Out
Goal: End session.
Flow:
- User selects ‚ÄúLog Out‚Äù
- Backend clears session token
- User returns to login screen


‚≠ê USER ROADMAP (Reader Role)
1. Library (Book Collection)
This is the first real feature users interact with.
Blueprint: users
Model: Library (user_id, book_id)
Routes:
- POST /users/me/library
Add a book to the user‚Äôs library.
- GET /users/me/library
View the user‚Äôs library.
- DELETE /users/me/library/<book_id> (optional)
Remove a book from the library.
Why this matters:
Users must own a book before they can create playlists for it.

2. Playlists (for books in their library)
Once a user has a book, they can create playlists for it.
Blueprint: playlists
Models:
- Playlist (id, user_id, book_id, title, description)
- PlaylistSong (playlist_id, song_id, order)
Routes:
- POST /playlists
Create a playlist for a book in the user‚Äôs library.
- GET /playlists/<id>
View a playlist.
- PUT /playlists/<id>
Update playlist details.
- DELETE /playlists/<id>
Delete playlist.
- POST /playlists/<id>/songs
Add a song to a playlist.
- DELETE /playlists/<id>/songs/<song_id>
Remove a song.
Why this matters:
This is the core of your app ‚Äî the ‚Äúplaylist for a book‚Äù experience.

3. Search (Books, Songs, Authors)
These are discovery features.
Blueprints:
- books
- songs
- authors
Routes:
- GET /books/search
- GET /books/<id>
- GET /songs/search
- GET /authors/search
- GET /authors/<id>
Why this matters:
Users need to find books ‚Üí add to library ‚Üí create playlists.

4. Apply for Author Role
This is how a user becomes an author.
Blueprint: users or authors
Model: AuthorApplication (user_id, status, notes)
Route:
- POST /users/apply-author
Why this matters:
This is the bridge between Reader ‚Üí Author.

5. View Profiles
Users should be able to view:
- their own profile
- other users‚Äô profiles (especially authors)
Blueprint: users
Routes:
- GET /users/me (already done in auth)
- GET /users/<id>
Why this matters:
Playlists, authors, and contributors all link back to user profiles.
